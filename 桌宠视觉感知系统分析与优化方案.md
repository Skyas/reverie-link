# 桌宠视觉感知系统 · 分析与优化方案

> **日期**：2026-04-24  
> **范围**：视觉感知层（屏幕捕获 → 差异检测 → VLM 分析 → 场景管理 → 发言决策）

---

## 一、执行摘要

当前桌宠的视觉感知系统架构思路正确，但存在**一个致命瓶颈**（截屏性能）和**三个结构性缺陷**（调度模型、差异检测精度、预筛逻辑）。

这些问题的综合表现是：**用户在激烈游戏或屏幕内容剧变时，桌宠反应延迟 2–5 秒，甚至完全无反应**，错失关键时刻。

本方案按模块拆解根因，给出**首选优化路径**（以效果优先，不限制工程量）和**精简次选方案**（1 天工作量内可获得 70% 提升）。

---

## 二、现状总评

| 维度 | 当前实现 | 成熟项目水平 | 差距 |
|------|---------|-----------|------|
| 屏幕捕获帧率 | PIL ImageGrab ≈ 2–3 FPS | DXGI / WGC ≈ 60–240 FPS | **50–100 倍** |
| 差异检测精度 | 6×4 分块 RGB 均值比较 | SSIM + 感知哈希 + 边缘检测 | 漏检 / 误检严重 |
| 捕获到 VLM 延迟 | 3–30 秒 + 截图 400 ms | 亚秒级 | 错失关键时刻 |
| 脏区域感知 | 无 | 仅捕获变化区域 | 全屏重复捕获浪费 |
| 键鼠预触发 | 仅影响 sleep 间隔 | 操作风暴立即截图 | 无法捕捉突发瞬间 |

## 三、根因详析（逐模块）

### 3.1 screen_capture.py —— 性能瓶颈的元凶

#### 问题一：PIL ImageGrab 是全系统最慢的截屏方式

多份开源项目实测数据（AutoDrive-GTA5、screen-capture-dataset、open-world-agents/desktop-env）一致表明：

| 方案 | 1920×1080 单帧耗时 | 等效帧率 |
|------|------------------|---------|
| PIL ImageGrab | 400–500 ms | **2 FPS** |
| MSS | 16–30 ms | 30–60 FPS |
| DXcam / d3dshot | 4–16 ms | 60–240 FPS |
| Windows Graphics Capture | < 1 ms（回调模式） | 跟随显示器刷新率 |

当前 `capture_screen()` 返回 PNG 字节流，后续流程中：
1. `compute_pixel_diff` 重新解码为 PIL Image
2. `compress_for_vlm` 再次解码并缩放到 960×540 JPEG

**同一张图被编码-解码-再编码至少 3 次**，每一步都损耗性能。

#### 问题二：Python 双重循环遍历像素

`compute_pixel_diff` 中的：

```python
for y in range(y0, y1):
    for x in range(x0, x1):
        p1 = pix1[x, y]
```

这种逐像素访问在 C 层只需几微秒，Python 层却需要 **10 ms+**。

#### 问题三：多显示器兼容性

PIL ImageGrab 在多显示器 / DPI 缩放环境下有已知 Bug（Pillow #7898），可能截取到错误屏幕或黑边。

---

### 3.2 capture_strategy.py —— 预筛逻辑与时机判断的矛盾

设计意图是"像素差只用来跳过纯静态帧"，但实际存在三重矛盾：

#### 矛盾一：STATIC_THRESHOLD = 0.5% 过低

战舰世界、原神、地平线等大面积同色场景的游戏，即使准星移动、远处炮弹落水、HUD 数值变化，6×4 分块的均值差异往往 **< 0.3%**。

**后果**：该发 VLM 的帧 → 被误判为 `static_frame` → 只加 1 分 → 桌宠"看不见"关键变化。

#### 矛盾二：boring_streak 惩罚机制

连续 3 次 VLM 返回 `interest_score <= 2` 就跳帧。但问题是：

- 对线期、走路探索等"无新事件但画面在动"的场景，VLM 给的低分是合理的
- 跳帧后，下一帧可能恰好是团战爆发，但因为 streak 被重置而错过
- 与像素阈值形成**双重漏检**

#### 矛盾三：is_first_frame 语义混淆

```python
is_first_frame=(pixel_diff >= 100.0)  # 首帧 pixel_diff 被设为 100
```

`pixel_diff >= 100` 恒为 True，但它同时承担了"首帧"和"有场景"两个判断。`has_current_scene` 才应该是判断是否有场景的关键，这个条件组合导致首帧后的策略不稳定。

---

### 3.3 像素差异算法 —— 该精细的地方太粗糙

当前算法流程：

1. 缩到 240×120
2. 分成 6×4 = 24 块
3. 每块求 RGB 均值
4. 比较欧氏距离
5. 权重：70% 最大块差异 + 30% 平均差异

**各场景表现**：

| 场景 | 实际变化 | 算法表现 | 结果 |
|------|---------|---------|------|
| MOBA 血量 100%→20% | HUD 数字变化 | 数字区域仅占一个块的 10%，均值变化 < 0.5% | **漏检** |
| FPS 准星对准不同背景 | 中心区域颜色偏移 | 单块均值偏移约 1–2% | 有时检出 |
| 视频暂停→播放 | 画面完全静止 | 均值差异 0% | 正确跳过 |
| 浏览器滚动页面 | 大面积内容位移 | 多个块均值剧变 | 正确检出 |
| 云层/光影飘过 | 全局亮度缓慢变化 | 多区块均值偏移 2–3% | **误检** |
| 加载画面转圈 | 小图标旋转 | 小区域周期性变化 | 不稳定 |

成熟项目（screen-capture-dataset、ShareX）使用的是 **SSIM（结构相似性指数）** 或 **感知哈希（pHash）**。SSIM 专门模拟人类视觉系统对亮度、对比度、结构的敏感度，比简单均值比较可靠得多。

---

### 3.4 vlm_client.py —— 增量分析未降低成本，反而增加风险

增量 prompt 设计本身很好，但存在两个问题：

**成本问题**：无论 `background` 还是 `incremental`，VLM 都需要接收**整张图片**。模型处理 960×540 图片的 token 消耗基本相同。省掉的只是输出 token（几百字），但视觉理解的核心成本（图片编码）一点没少。

**准确性问题**：增量分析反而可能降低准确性。VLM 看单帧判断"和上一帧差不多"是困难的，因为：
- 压缩后的 JPEG 本身就损失了大量细节
- 模型没有"上一帧"的视觉记忆，只有文本描述（`prev_descriptions`）
- 所以它只能基于你给的文字 + 当前图猜测，准确率不高

---

### 3.5 event_buffer.py —— 去重逻辑过于粗暴

`_is_similar` 使用 2-gram Jaccard 相似度 > 70% 判定重复：

```
上一帧："血量约30%，正在和敌人交火"
当前帧："血量约20%，正在和敌人交火"
```

这两个描述的 2-gram 重合度极高（约 85%），会被判定为"相似" → **只加分不加事件**。

**后果**：
- 桌宠的上下文里永远看不到"血量下降"这个关键事实
- 用户从满血打到残血，event_buffer 只累积分数，没有记录变化轨迹
- 发言时拿到的 `context_prompt` 还是旧的满血描述

---

### 3.6 vision_system.py + activity_monitor.py —— 调度耦合问题

当前调度链条：

```
activity_monitor.get_adaptive_interval() → asyncio.sleep(interval) → _process_frame()
```

这是**单线程顺序调度**：先睡 N 秒，再截图，再判断，再调 VLM，再睡 N 秒。

**问题**：
- **sleep 期间发生的画面剧变完全丢失**。比如 sleep 最后一秒刚好是五杀时刻，睡完去截图，画面已经切到结算界面了
- **键鼠活动和画面变化没有关联分析**。键鼠 `INTENSE` 状态只影响 sleep 时长，不触发"立即截图"
- 没有对"键鼠活动突增"做预触发（proactive capture）

成熟项目的做法是：**捕获线程独立以最高帧率运行，分析线程异步消费，根据活动状态选择消费频率**。

---

### 3.7 scene_manager + game_detector —— 场景切换反应太慢

`needs_full_analysis` 只在以下情况触发：

1. 首帧
2. 窗口标题变化
3. pixel_diff > 50%

**游戏内场景切换**（从对线进入团战、从菜单进入游戏、从存活变为死亡观战）往往不涉及窗口标题变化，`pixel_diff` 也可能 **< 50%**（因为还是同一款游戏的渲染风格）。

`vlm_result.scene_changed` 字段虽然存在，但它是在**已经调用 VLM 之后**才得到的。为了知道场景变化，已经付出了一次完整 VLM 调用。

### 5.2 阶段 2：差异检测重构（screen_capture.py）

#### 首选方案：SSIM + 感知哈希 + 边缘差异 三重检测

```python
import cv2
import numpy as np
from skimage.metrics import structural_similarity as ssim
import imagehash
from PIL import Image

def compute_frame_diff(img1_bytes: bytes, img2_bytes: bytes) -> dict:
    # 统一转为 numpy
    img1 = cv2.imdecode(np.frombuffer(img1_bytes, np.uint8), cv2.IMREAD_COLOR)
    img2 = cv2.imdecode(np.frombuffer(img2_bytes, np.uint8), cv2.IMREAD_COLOR)

    # 1. SSIM（结构相似性）
    gray1 = cv2.cvtColor(img1, cv2.COLOR_BGR2GRAY)
    gray2 = cv2.cvtColor(img2, cv2.COLOR_BGR2GRAY)
    gray1 = cv2.resize(gray1, (256, 144))
    gray2 = cv2.resize(gray2, (256, 144))
    ssim_score, _ = ssim(gray1, gray2, full=True)

    # 2. 感知哈希（pHash）
    pil1 = Image.fromarray(cv2.cvtColor(img1, cv2.COLOR_BGR2RGB))
    pil2 = Image.fromarray(cv2.cvtColor(img2, cv2.COLOR_BGR2RGB))
    phash1 = imagehash.phash(pil1, hash_size=16)
    phash2 = imagehash.phash(pil2, hash_size=16)
    phash_diff = phash1 - phash2

    # 3. 边缘差异（Canny）
    edges1 = cv2.Canny(gray1, 50, 150)
    edges2 = cv2.Canny(gray2, 50, 150)
    edge_diff = np.sum(cv2.absdiff(edges1, edges2)) / (edges1.size * 255)

    # 4. 区域加权差异
    h, w = gray1.shape
    weights = np.ones_like(gray1, dtype=np.float32)
    weights[h//4:3*h//4, w//4:3*w//4] *= 2.0  # 中心权重高
    # 四角 HUD 区域...
    weighted_mse = np.average((gray1.astype(np.float32) - gray2.astype(np.float32))**2, weights=weights)

    return {
        "ssim": ssim_score,
        "phash": phash_diff,
        "edge_diff": edge_diff,
        "weighted_mse": weighted_mse,
        "has_significant_change": ssim_score < 0.95 or phash_diff > 8 or edge_diff > 0.05,
    }
```

**各指标特性**：

| 指标 | 对什么敏感 | 对什么鲁棒 |
|------|-----------|-----------|
| SSIM | 结构性变化（物体出现/消失、场景切换） | 亮度偏移、轻微压缩 |
| pHash | 布局变化、大幅内容改变 | 亮度、对比度、压缩 |
| Edge diff | UI/HUD 变化、文字变化、准星移动 | 纯色背景飘动 |
| 加权 MSE | 重点区域变化 | 边缘非重要区域变化 |

#### 次选方案：保留分块框架，向量化 + 增加权重

```python
def compute_pixel_diff_fast(img1_bytes: bytes, img2_bytes: bytes) -> float:
    import numpy as np
    from PIL import Image

    img1 = Image.open(BytesIO(img1_bytes)).convert('RGB').resize((320, 180))
    img2 = Image.open(BytesIO(img2_bytes)).convert('RGB').resize((320, 180))

    a1 = np.array(img1, dtype=np.float32)
    a2 = np.array(img2, dtype=np.float32)

    h, w = 180, 320
    grid_h, grid_w = 6, 8
    block_h, block_w = h // grid_h, w // grid_w

    b1 = a1.reshape(grid_h, block_h, grid_w, block_w, 3)
    b2 = a2.reshape(grid_h, block_h, grid_w, block_w, 3)

    mean1 = b1.mean(axis=(1, 3, 4))
    mean2 = b2.mean(axis=(1, 3, 4))

    diff_map = np.sqrt(((mean1 - mean2)**2).sum(axis=-1))

    # 中心权重高
    weights = np.ones_like(diff_map)
    cy, cx = grid_h // 2, grid_w // 2
    weights[cy-1:cy+2, cx-1:cx+2] *= 2.0

    weighted_max = np.max(diff_map * weights)
    weighted_avg = np.average(diff_map, weights=weights)
    return weighted_max * 0.7 + weighted_avg * 0.3
```

### 5.3 阶段 3：调度模型重构（vision_system.py + activity_monitor.py）

#### 首选方案：生产者-消费者异步模型 + 键鼠预触发

```python
class VisionSystem:
    def __init__(self, speech_queue: asyncio.Queue):
        self._capture_queue = asyncio.Queue(maxsize=5)
        self._vlm_analysis_queue = asyncio.Queue()

    def start(self):
        asyncio.create_task(self._capture_loop(), name="capture")
        asyncio.create_task(self._filter_loop(), name="filter")
        asyncio.create_task(self._vlm_consumer_loop(), name="vlm_consumer")

    async def _capture_loop(self):
        """固定高频率捕获，轻量判断"""
        while True:
            frame = await asyncio.to_thread(capture_screen)
            if frame is None:
                await asyncio.sleep(0.1)
                continue

            if self._prev_frame is not None:
                is_changed = await asyncio.to_thread(self._quick_changed, self._prev_frame, frame)
                if not is_changed:
                    self._prev_frame = frame
                    continue

            # 队列满时丢弃旧帧
            if self._capture_queue.full():
                try:
                    self._capture_queue.get_nowait()
                except:
                    pass
            await self._capture_queue.put((time.time(), frame))
            self._prev_frame = frame
            await asyncio.sleep(self._get_capture_interval())

    async def _filter_loop(self):
        """智能筛选，批量丢弃相似帧"""
        while True:
            ts, frame = await self._capture_queue.get()
            while not self._capture_queue.empty():
                _, newer_frame = await self._capture_queue.get()
                frame = newer_frame
            await self._vlm_analysis_queue.put(frame)

    async def _vlm_consumer_loop(self):
        """异步 VLM 分析"""
        while True:
            frame = await self._vlm_analysis_queue.get()
            await self._process_frame(frame)
```

### 5.4 阶段 4：VLM 调用策略优化（capture_strategy.py）

#### 首选方案：引入分层 VLM 策略

```python
class CaptureStrategy:
    QUICK_REJECT_SSIM = 0.98      # 几乎没变，直接跳过
    LOCAL_SCREEN_SSIM = 0.90      # 有变化但不大
    FULL_VLM_SSIM = 0.90           # 需要深入分析

    async def should_call_vlm(self, diff_result: dict) -> tuple:
        ssim = diff_result["ssim"]

        if ssim > self.QUICK_REJECT_SSIM:
            return False, "no_perceptual_change"

        if ssim > self.LOCAL_SCREEN_SSIM:
            is_interesting = await self._local_classifier(diff_result)
            if not is_interesting:
                return False, "local_filtered"

        return True, "significant_change"
```

#### 次选方案：引入 CLIP 本地轻量预筛

```python
import clip
import torch
from PIL import Image

class LocalVisualFilter:
    def __init__(self):
        self.model, self.preprocess = clip.load("ViT-B/32", device="cpu")
        self.prev_features = None

    @torch.no_grad()
    def is_interesting(self, img_bytes: bytes) -> bool:
        img = Image.open(BytesIO(img_bytes)).convert("RGB")
        img = self.preprocess(img).unsqueeze(0)
        features = self.model.encode_image(img)
        features /= features.norm(dim=-1, keepdim=True)

        if self.prev_features is None:
            self.prev_features = features
            return True

        similarity = (features @ self.prev_features.T).item()
        self.prev_features = features * 0.7 + self.prev_features * 0.3
        return similarity < 0.92
```

### 5.5 阶段 5：场景与游戏检测增强

#### scene_manager.py 增强

```python
class SceneManager:
    def needs_full_analysis(self, window_title: str, pixel_diff: float, process_info: dict) -> bool:
        # 原有条件
        if self._current_result is None:
            return True
        if window_title != self._last_window_title:
            return True
        if pixel_diff > 50.0:
            return True

        # 新增：进程变化检测
        proc_name = process_info.get("process_name", "")
        if proc_name != self._last_process_name:
            self._last_process_name = proc_name
            return True

        # 新增：窗口标题模式匹配
        title_lower = window_title.lower()
        if "league" in title_lower or "lol" in title_lower:
            if self._current_result.scene_type != "game":
                return True

        return False
```

### 5.6 阶段 6：event_buffer 去重逻辑优化

#### 首选方案：语义 + 关键事实驱动去重

```python
class EventBuffer:
    def _is_semantic_duplicate(self, new_desc: str, last_desc: str,
                               new_facts: list, last_facts: list) -> bool:
        # 1. 关键事实不同 → 不算重复
        if new_facts and last_facts:
            fact_sim = len(set(new_facts) & set(last_facts)) / max(len(set(new_facts)), len(set(last_facts)))
            if fact_sim < 0.5:
                return False

        # 2. 描述相似度
        if self._text_similarity(new_desc, last_desc) > 0.75:
            return True

        return False
```

## 七、精简次选方案（1 天工作量 / 70% 效果提升）

如果暂时不想大改，**只做三件事**：

### 改动 1：截屏引擎替换

`screen_capture.py` 中 `PIL.ImageGrab` 换成 `mss`，`compute_pixel_diff` 改用 numpy 向量化（见 5.2 次选方案）。

### 改动 2：捕获与 VLM 解耦

`vision_system.py` 的 `_run_loop` 中，捕获和 VLM 调用不要互相阻塞。VLM 分析放在独立协程，主循环只负责把帧放入队列。

### 改动 3：放宽预筛阈值

`capture_strategy.py` 中：
- `STATIC_THRESHOLD` 从 0.5% 提高到 3%
- 添加规则：键鼠处于 `INTENSE` 状态时，忽略静态阈值，强制捕获

**这三项合计不超过 1 天工作量，但能让"错失关键时刻"的问题基本解决。**
